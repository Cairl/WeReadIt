## 架构评审 WeReadIt 项目

通读 main.py / config.py / exchange.py / push.py / log_utils.py / deploy.yml 后，输出完整架构分析报告到 `docs/architecture-review.md`。

### 关键发现

**P0 问题（影响功能正确性）**：
- `.github/workflows/deploy.yml` 漏传 `WEREAD_ACCESS_TOKEN` / `WEREAD_PLATFORM` / `EXCHANGE_AWARD` 三个环境变量，导致 CI 中兑换功能静默失效（token 为空直接跳过）
- `main.py` 的 `refresh_cookie()` 在 while 循环内被调用后不重试当前阅读请求，会丢失一次阅读进度，且 `lastTime` 未更新导致下次 `rt` 异常变大
- `github_sync.bat` 是另一个项目（GitHubSync）的入口，不属于本仓库

**结构性问题**：
- `config.py` 单文件 4 项职责（环境变量/业务数据/curl解析/顶层副作用）
- `main.py` 是脚本而非模块，顶层直接执行副作用，无法 import 单测
- `push.py` 的 `PushNotification` 把 4 渠道塞一起，4 个 `push_xxx` 方法重试逻辑完全重复
- `exchange.py` 相对规范但 `_query_awards` 和 `_exchange_single_award` 重复
- 无 `requirements.txt` / `pyproject.toml` / lint / 测试 / 类型注解
- `requests` 未用 Session，每次新建 TCP 连接

### 推荐目录结构

采用 src layout + 分层（core/infra/push/utils/schemas），把扁平 6 文件拆为 ~20 个职责单一的模块。push 用策略模式 + 注册表，新增渠道只新建 1 个文件 + 1 行注册。

### 重构分 5 阶段

1. 基础设施补全（requirements/pyproject/ruff/修 deploy.yml/删 github_sync.bat）
2. 配置中心化（拆 config.py 为 config/constants/schemas/books.json/infra/curl_parser）
3. main.py 模块化（提取 reader/crypto/app/__main__）
4. push 策略化（base/registry/4 个渠道文件）
5. exchange 重构 + HttpClient 抽象 + 单元测试

每阶段独立可验证，业务逻辑零中断。

报告路径：`docs/architecture-review.md`

### 关键设计决策
- 保留根目录 `main.py` 作为兼容入口（CI 与本地用户无需改命令）
- `Config` 用 `frozen=True` dataclass，防止运行时被意外修改
- `Pusher` 持有 `cfg` 引用而非只 token，让 Telegram 能取到 chat_id
- `registry.py` 末尾显式 import 4 个渠道模块触发注册（用 noqa 标注 E402）
- `exchanger.py` 重试改为指数退避：`RETRY_INTERVAL * (2 ** attempt)`
- HttpClient 提供 `update_cookie()` 方法，cookie 刷新后更新 Session 而非替换整个 cookies 字典

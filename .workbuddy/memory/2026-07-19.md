## 架构评审 WeReadIt 项目

通读 main.py / config.py / exchange.py / push.py / log_utils.py / deploy.yml 后，输出完整架构分析报告到 `docs/architecture-review.md`。

### 关键发现

**P0 问题（影响功能正确性）**：
- `.github/workflows/deploy.yml` 漏传 `WEREAD_ACCESS_TOKEN` / `WEREAD_PLATFORM` / `EXCHANGE_AWARD` 三个环境变量，导致 CI 中兑换功能静默失效（token 为空直接跳过）
- `main.py` 的 `refresh_cookie()` 在 while 循环内被调用后不重试当前阅读请求，会丢失一次阅读进度，且 `lastTime` 未更新导致下次 `rt` 异常变大
- `github_sync.bat` 是另一个项目（GitHubSync）的入口，不属于本仓库

**结构性问题**：
- `config.py` 单文件 4 项职责（环境变量/业务数据/curl解析/顶层副作用）
- `main.py` 是脚本而非模块，顶层直接执行副作用，无法 import 单测
- `push.py` 的 `PushNotification` 把 4 渠道塞一起，4 个 `push_xxx` 方法重试逻辑完全重复
- `exchange.py` 相对规范但 `_query_awards` 和 `_exchange_single_award` 重复
- 无 `requirements.txt` / `pyproject.toml` / lint / 测试 / 类型注解
- `requests` 未用 Session，每次新建 TCP 连接

### 推荐目录结构

采用 src layout + 分层（core/infra/push/utils/schemas），把扁平 6 文件拆为 ~20 个职责单一的模块。push 用策略模式 + 注册表，新增渠道只新建 1 个文件 + 1 行注册。

### 重构分 5 阶段

1. 基础设施补全（requirements/pyproject/ruff/修 deploy.yml/删 github_sync.bat）
2. 配置中心化（拆 config.py 为 config/constants/schemas/books.json/infra/curl_parser）
3. main.py 模块化（提取 reader/crypto/app/__main__）
4. push 策略化（base/registry/4 个渠道文件）
5. exchange 重构 + HttpClient 抽象 + 单元测试

每阶段独立可验证，业务逻辑零中断。

## 上游项目 wxread 保活策略深度分析

完整阅读 findmover/wxread 4 个 Python 文件 + 1 个 workflow,输出保活策略分析报告到 `wxread_keepalive_analysis.md`。

### 核心结论

WeReadIt 是 wxread 的重构,但 wxread 中至少 13 处保活策略必须 1:1 迁移,不能"优化":

1. **启动强制 refresh_cookie** (main.py:76) — 上线握手,不能删
2. **3 种 COOKIE_DATA_VARIANTS** (main.py:19) — 接口兼容保险,不能简化为 1 种
3. **`fix_no_synckey()`** (main.py:58-59) — 状态同步修复,看似无用实为关键
4. **bookId `"3300060341"`** — 不在 book 列表里,可能是上下文重建触发器
5. **`lastTime = now - 30`** (main.py:78) — 伪造初始阅读时长
6. **`data.pop('s')`** — 防止用旧签名
7. **data 固定字段** (ci/co/sm/pr/ps/pc) — 作者实测得出"必须固定",改动会导致时长不增加
8. **`time.sleep(30)`** — 真实阅读节奏,调快会触发风控
9. **`ts` jitter + `rn` 随机** — 风控规避
10. **`baggage` Sentry 头** — 浏览器指纹,不能删
11. **keepalive-job** — 防 GitHub Actions 60 天自动禁用
12. **DNS 8.8.8.8** — 基础设施保活
13. **push 5 次重试 + 180~360s 抖动** — 通知层保活

### 关键发现

- `fix_no_synckey` 调用 chapterInfos 接口,bookId 是 `3300060341`(不在 book 列表里),作者刻意保留这个写死的 ID,长期删除会导致"刷了但时长不增加"
- `data` 字段中 ci/co/sm/pr/ps/pc 不更新看似违反真实阅读,实则是作者实测得出的"必须固定"上下文指纹
- 失败重试时"不 sleep 不递增 index"是正确设计(本次重试不计入进度),不是 bug
- README 第 76 行明确警告"建议保留 data 字段"——这是作者从 issue 反馈得到的经验

## WeReadIt 保活策略改进计划

审计 WeReadIt 当前实现 vs wxread 13 类保活经验,输出改进计划到 `wxread_keepalive_improvement_plan.md`。

### 审计结论

**21 项保活策略全部 1:1 保留**(很好),但重构引入 3 个新风险:

1. **P0.1 requests.Session 自动合并 Set-Cookie** — `HttpClient` 用 Session 管理 cookies,服务器响应的 Set-Cookie 会自动覆盖业务层 `wr_skey[:8]` 截断。原版用 `requests.post(cookies=cookies)` 显式传不会回写。修复:Session 保留 TCP 复用,但 cookies 改为业务层独占字典,每次 post 显式传
2. **P0.2 headers 双重传递** — `app.py` 构造 HttpClient 传一次 `cfg.headers`,`reader.py` 每次请求又传一次。建议统一只在构造时传
3. **P0.3 无熔断机制** — 连续无 synckey 或连续 cookie 失败会死循环(GitHub Actions 6h 超时)。建议加 `MAX_NO_SYNCKEY=5` / `MAX_COOKIE_FAIL=3` 熔断

### 待优化项(7 项)

- P1.1 fix_no_synckey 后不重试 read,会丢失本次进度
- P1.2 refresh_cookie 失败立即抛异常,无指数退避(且 `except Exception` 太宽,违反 AGENTS.md)
- P1.3 请求超时统一 10s,read 应 15s
- P1.4 sleep(30) 无 jitter,可加 ±2s
- P2.1 缺少运行 metrics(synckey 成功率/fix 触发次数)
- P2.2 保活策略未文档化,新人易误删
- P3.1 缺少保活策略单元测试
- P3.2 默认 cookies 含敏感字段,未配 curl_bash 时应报错

### 关键认知

- WeReadIt 重构整体优秀,21 项保活策略完整保留,包括混淆变量名 `_7032f5`/`_cc1055`/`_19094e`、bookId `"3300060341"`、data 固定字段
- 但 `requests.Session` 的 Cookie 自动合并是重构副作用,原版用全局 `requests.post` 不会有此问题
- 改进原则:**保活策略 1:1 迁移是底线,任何"优化"都必须先证明不破坏保活机制**
